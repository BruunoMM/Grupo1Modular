83  * - verificar se todos os parâmetros foram lidos

85  * por exemplo: verificar se os índices de acesso a elementos

139 /***********************************************************************
140 *
141 * $FC Função: TSTE &Efetuar comandos de teste específicos
142 *
143 * $ED Descrição da função
144 * Efetua os diversos comandos de teste específicos para o módulo
145 * sendo testado.
146 *
147 * Cada módulo de teste específico deve desenvolver esta função.
148 * Poderá conter diversas outras conforme necessário. No entanto
149 * estas devem todas ser funções encapsuladas no módulo (static)
150 *
151 * Freqüentemente é necessário desenvolver um contexto para o teste.
152 * Por exemplo, pode ser necessário tratar n elementos disponibilizados
153 * pelo módulo sob teste. Neste caso deveria ser criado um vetor
154 * global com referências para estas elemento. Ao iniciar os teste
155 * este vetor deverá ser inicializado. Ao terminar, os elementos do
156 * vetor referenciando algum elemento devem ser destruídos.
157 *
158 * Uma das formas para estabelecer o contexto seria a de incluir
159 * no mçodulo de teste específico comando semelhantes a:
160 *
161 * =iniciaXXX que inicializa o contexto do módulo sob teste para
162 * vazio. Este comando deve ser utilizado uma única vez
163 * logo ao inciar a execução do teste.
164 * =reset restaura o contexto para vazio. Este comando usualmente
165 * assumirá que o contexto a esvaziar esteja correto.
166 * O comando deverá liberar os espaços alocados aos
167 * elementos, anular as referências a ele, inicializar
168 * valores de vetores de referências, etc.
169 * O comando tem utilidade especial ao segmentar um teste
170 * em partes estanques e ao controlar vazamento de
171 * memória.
172 * =terminaXXX este comando destrói todo o contexto e não o
173 * reinicializa. O objetivo é poder verificar vazamento
174 * de memória.
175 *
176 * Evidentemente, outra forma seria a de alterar as funções de
177 * inicialização e término do módulo de teste genérico o que é pouco
178 * recomendável pois viola o princípio de um mesmo módulo genérico
179 * servir para testar todos os construtos.
180 *
181 * Para cada função do módulo a ser testado deve existir pelo menos
182 * um comando. Em alguns casos pode ser interessante criar
183 * comandos que armazenem, alterem ou consultem variáveis de
184 * estado do próprio módulo de teste. Ver os comentários sobre
185 * contexto acima.
186 *
187 * Todos os controles de consistência devem ser realizados no
188 * módulo de teste específico.
189 *
190 * A função TST_EfetuarComando( ) deve retornar as condições de retorno
191 * identificadas no tipo TST_tpCondRet definida no módulo TesteGen.
192 *
193 * As funções do módulo sob teste que retornam algum valor sempre
194 * devem sempre ter este valor verificado, mesmo sendo uma condição de
195 * retorno. Para fazer a vereficação da corretude, recomenda-se
196 * utilizar as funções TST_CompararXXX definidas no módulo genérico.
197 * Assegura-se assim a uniformidade da saída gerada durante o teste.
198 *
199 * Para realizar a análise de um comando de teste recomenda-se
200 * utilizar as funções definidas no módulo LerParm. Elas, além
201 * de realizarem a análise léxica, permitem o uso intercambiável
202 * de parâmetros simbólicos ou literais. Também realizam a
203 * verificação da corretude dos tipos dos parâmetros, contribuindo
204 * assim para a redução de erros de redação de scripts de teste
205 * que possam passar despercebidos.
206 *
207 * As funções a serem testadas que requeiram algum argumento, devem
208 * ser chamadas com valores (parâmetros) extraídos do comando de
209 * teste sendo executado.
210 *
211 * $EP Parâmetros
212 * $P ComandoTeste - String contendo o comando
213 *
214 * $FV Valor retornado
215 * TST_CondRetXXX - veja a descriÓcão dos possíveis valores
216 * retornados na especificação do correspondente
217 * enum
218 *
219 ***********************************************************************/
220
